1. The Platform in Context
This is a stock market simulation for a college event. It runs for roughly four hours. About 100 to 150 students participate, trading virtual shares across seven rounds while reacting to news events injected by an admin team. The platform exists to create the experience — drama, strategy, learning — not to model financial markets with academic precision.
That context should drive every decision. The platform that needs to survive years of production load, regulatory scrutiny, and distributed teams is a different system entirely. This one needs to start reliably, stay up for four hours, and be operable by a small admin team who are simultaneously hosting an event. Any complexity beyond that is a liability, not an asset.

The Guiding Question
Before adding anything to this system, ask: does the event break without it? If the answer is no, leave it out. The event breaks if trading stops. It does not break if the price engine lacks GARCH volatility clustering.

The previous outputs on this platform explored two extremes: an over-engineered distributed system with Redis, BullMQ, Cholesky decompositions, and Grafana dashboards; and an aggressively stripped-down sketch that left out things that genuinely matter. Neither is right. This document synthesises both into a system that is minimal but not naive — one that eliminates real complexity while preserving real reliability.
What was correctly eliminated: Redis, BullMQ, Docker, distributed locks, PgBouncer, Next.js App Router, uWebSockets, Prometheus, Grafana, monorepo tooling, saga patterns, Cholesky sector correlation, GARCH volatility, and the 14-layer trade execution flow. These add weeks of build time and operational complexity that serve a system operating at a scale this event will never reach.
What was incorrectly eliminated: Proper session persistence so a server restart does not log out 150 participants mid-event, a bounded price engine so stocks do not drift to zero, a database-level race condition guard on the last share, and a one-click emergency halt for the admin. These are not over-engineering. They are the difference between an event that runs smoothly and one that requires emergency intervention.

2. The Four Things That Must Work
Everything else in this document exists to make these four things reliable. If any of them fails during the event, the event fails. If all four work, the event works regardless of what optional features are missing.

The price engine is a setInterval inside the main process. It runs every 10 seconds, updates prices in the database and in an in-memory map, and emits a single Socket.IO broadcast. There is no separate process, no message queue, and no pub/sub infrastructure. If the server restarts, the price engine re-seeds the in-memory map from the database on startup. The gap is at most 10 seconds.
One intentional omission: Redis. Sessions live in PostgreSQL via connect-pg-simple. Live prices live in a JavaScript object in memory, updated by the price engine. This object is the authoritative price source for all trade executions. No Redis lock is needed because Node.js is single-threaded — one async function runs at a time. The database transaction handles concurrent correctness.

4.2 The One Query That Must Be Correct
Every other query is straightforward. This one is not. The trade execution must use a serializable transaction with a row-level lock to handle the case where two participants simultaneously buy the last available share. This is not over-engineering — it is the minimum correct implementation.
-- Buy order inside a single serializable transaction
BEGIN;

SELECT available_supply, current_price
FROM stocks WHERE id = $stockId
FOR UPDATE;
-- This lock prevents any other transaction from reading or writing
-- the supply until this transaction commits or rolls back.

-- All validation happens inside the lock:
--   available_supply >= quantity
--   user.cash_balance >= total_cost
--   stock.is_tradeable = true
--   round.status = active

UPDATE stocks SET available_supply = available_supply - $qty  WHERE id = $stockId;
UPDATE users  SET cash_balance     = cash_balance - $cost    WHERE id = $userId;
INSERT INTO trades (user_id, stock_id, side, quantity, price, fee) VALUES (...);
INSERT INTO holdings (user_id, stock_id, quantity, avg_buy_price)
  VALUES ($userId, $stockId, $qty, $price)
  ON CONFLICT (user_id, stock_id)
  DO UPDATE SET
    quantity      = holdings.quantity + $qty,
    avg_buy_price = (holdings.quantity * holdings.avg_buy_price + $qty * $price)
                    / (holdings.quantity + $qty);
COMMIT;

With FOR UPDATE on the stock row, the second concurrent buyer waits until the first transaction commits, then re-reads the now-reduced supply. If it is zero, the transaction fails cleanly with an insufficient supply error. No Redis locks, no distributed coordination — just a database doing what databases are built to do.

5.2 Price Engine
Simple, bounded, and adequate. The engine moves each tradeable stock by a small random amount within its volatility band every 10 seconds. A soft mean-reversion nudge prevents long-run drift without any parameter calibration. Hard floor and ceiling limits mean prices can never collapse or explode regardless of what events the admin triggers.
const livePrices = {};  // seeded on startup from DB

async function priceTick() {
  const stocks = await prisma.stock.findMany({ where: { is_tradeable: true } });
  for (const stock of stocks) {
    const band  = stock.volatility_pct / 100;
    const noise = (Math.random() * 2 - 1) * band;

    // Soft mean-reversion: gentle pull toward base_price if drifted > 60%
    const drift = (stock.current_price - stock.base_price) / stock.base_price;
    const pull  = Math.abs(drift) > 0.6 ? -drift * 0.05 : 0;

    const raw      = stock.current_price * (1 + noise + pull);
    const floor    = stock.base_price * 0.20;   // never below 20% of base
    const ceiling  = stock.base_price * 6.00;   // never above 600% of base
    const newPrice = Math.max(floor, Math.min(ceiling, raw));

    livePrices[stock.ticker] = newPrice;
    await prisma.stock.update({
      where: { id: stock.id }, data: { current_price: newPrice }
    });
  }
  io.emit('price_update', livePrices);  // one object, all tickers
}

setInterval(priceTick, 10_000);

The hard floor and ceiling eliminate the need for pre-event calibration. The soft mean-reversion handles slow drift across seven rounds. The engine is 20 lines of code and requires no tuning.

5.3 Trade Execution
The trade handler validates the request shape, checks round state from the in-memory round object, then runs the database transaction from Section 4.2. On success, it emits a portfolio_update to the participant's private Socket.IO room. It also increments a demand signal (a simple buy/sell count per ticker) that the price engine reads on the next tick to add a small directional nudge.

5.4 Session Strategy
Sessions are stored in PostgreSQL using connect-pg-simple. This is five lines of setup and the single most important reliability decision in the stack. In-memory sessions vanish when the server restarts. With database-backed sessions, all 150 participants stay logged in through any restart, deployment, or crash recovery. On an event where even one unplanned restart is disruptive, this matters.
const session   = require('express-session');
const pgSession = require('connect-pg-simple')(session);

app.use(session({
  store:             new pgSession({ conString: process.env.DATABASE_URL }),
  secret:            process.env.SESSION_SECRET,
  resave:            false,
  saveUninitialized: false,
  cookie:            { maxAge: 8 * 60 * 60 * 1000 }  // 8 hours
}));


6. Real-Time Layer
6.1 Socket.IO Setup
Three rooms cover every communication pattern. No custom namespace logic, no complex room management.
io.on('connection', (socket) => {
  const user = socket.request.session?.user;
  if (!user) { socket.disconnect(); return; }

  socket.join(`user:${user.id}`);  // private: portfolio updates
  socket.join('global');           // broadcast: prices, news, round status
  if (user.role === 'admin') socket.join('admin');

  // State sync on every connection -- this IS the reconnection strategy
  socket.emit('state_sync', {
    prices:             livePrices,
    round:              currentRound,
    leaderboardVisible: leaderboardRevealed,
  });
});


Reconnection is automatic
The state_sync emit fires on every connection, not just the first. When a participant's browser reconnects after a WiFi hiccup, they immediately receive current prices, round status, and leaderboard visibility. No reconnection-specific code path is needed.

6.2 Price Broadcast
Prices are broadcast as a single flat object: { VNTCH: 482.5, VNDFN: 315.0, ... }. The client replaces its entire local price map on every tick. This eliminates stale state bugs and makes portfolio value calculation trivial: iterate holdings and multiply by livePrices[ticker]. When a new IPO stock is added mid-event, the broadcast automatically includes it.

6.3 Event Broadcasts
When the admin triggers a news event, the server does three things: inserts the news record, applies price impacts to affected stocks in both the database and the livePrices map, then emits one Socket.IO event carrying both the news item and the updated prices. Participants see the headline appear and prices jump in the same render cycle.
// POST /api/admin/news
const { headline, detail, impacts } = req.body;
// impacts: [{ ticker: 'VNTCH', magnitude: 0.15 }, ...]

for (const { ticker, magnitude } of impacts) {
  livePrices[ticker] = livePrices[ticker] * (1 + magnitude);
  await prisma.stock.update({
    where: { ticker }, data: { current_price: livePrices[ticker] }
  });
}

await prisma.newsEvent.create({ data: { headline, detail, roundId } });

// One emit: updated prices + the news item
io.to('global').emit('news_event', { headline, detail, prices: livePrices });

8. Build Plan
Four weeks. The fourth week is explicitly a buffer and rehearsal week, not a development week. If the core is not complete and rehearsed by end of Week 3, the event is at risk. Treat that deadline as fixed.

8.1 Week 1: Core Trading
Goal: 
	•	Day 1: Scaffold Express server. Write Prisma schema. Connect to Supabase. Run prisma db push. Seed 15 stocks and 10 test users with bcrypt-hashed passwords.
	•	Day 2: Auth endpoints and session middleware with connect-pg-simple. Verify that a server restart does not destroy sessions.
	•	Day 3: Buy and sell endpoints with the serializable transaction. Manually test: two tabs simultaneously buying the last 5 shares of one stock. Only the available quantity should be sold in total.
	•	Day 4: Portfolio and stock list endpoints. Seed livePrices from the database on startup.
	•	Day 5: Basic frontend. Login screen. Stock list. Trade widget calling the endpoints. Portfolio display. No real-time yet.

Validation Gate - Week 1
Admin imports a CSV of 10 participants. Each logs in. One buys 100 VNTCH shares: cash decreases, holding appears. They sell 50: cash increases. The admin sees this in the participant monitor.

Goal: 
	•	Day 1: Socket.IO setup. Server rooms, state_sync on connection, client connection hook.
	•	Day 2: Price engine with setInterval. livePrices map updated each tick. io.emit('price_update') every 10 seconds. Client updates display and recomputes portfolio value.
	•	Day 3: Round control endpoints. Start emits round_status, stops emit round_status. Participant UI shows current round and open/closed indicator.
	•	Day 4: News trigger and direct shock in admin. Client news feed appends items. Chart ring buffer updates on price_update.
	•	Day 5: Polish participant dashboard. Stock list colour flashes on price change. Trade widget feels instant with optimistic UI. Chart scrolls smoothly.

Validation Gate - Week 2
Five browser tabs as participants, one as admin. Admin starts Round 1. All participant tabs show "Round 1 - Trading Open." Prices move every 10 seconds in sync. Admin triggers news: all tabs show headline, affected stocks jump simultaneously.

8.3 Week 3: Admin Panel and Deploy
Goal: 
	•	Day 1: Complete admin panel: news trigger with preview, free broadcast, emergency halt, all controls from Section 7.3.
	•	Day 2: Leaderboard. Computed in memory every 30 seconds. Revealed on admin command. Public display screen built.
	•	Day 3: Deploy to Render and Vercel. Set all environment variables. Verify end-to-end on live URLs, not localhost. Set up UptimeRobot health check.

	•	Prices look wrong: Use Direct Shock to correct any stock. 10 seconds.
	•	Participant cannot log in: Open Supabase SQL editor. Run: UPDATE users SET password_hash = 'new_hash' WHERE username = '...'. Or create a spare account in advance.
	•	Server restarts unexpectedly: Sessions are in PostgreSQL. Participants stay logged in. Price engine restarts automatically. One 10-second price gap. No action needed.
	•	A trade was clearly incorrect: Open Supabase SQL editor. Manually adjust cash_balance and delete the trade record. Takes 2 minutes.
	•	Something is badly broken: Click HALT TRADING immediately. Announce a brief technical pause. Fix the issue. Resume. A 5-minute pause is forgotten. A bad trade that cannot be undone is not.

11. Non-Obvious Improvements
These decisions do not add visible complexity but prevent real failures. Each takes under an hour to implement.

1. Database-backed sessions prevent a catastrophic restart
Covered in Section 5.4. Repeated here because it is commonly omitted and the consequences are severe: 150 participants simultaneously logged out mid-event. connect-pg-simple is five lines of code.

2. Hard price limits eliminate calibration
Built into the price engine: no stock can fall below 20% of its base price or rise above 600%. This means the admin team does not need to tune volatility parameters before the event or monitor for price runaway during it. The engine is self-bounding.

3. The free broadcast replaces five optional features
Leaked intelligence, analyst reports, board announcements, voting outcomes, rumour rounds — every one of these originally required custom mechanics. They all reduce to the same action: show everyone some text and let them react. The admin's free broadcast textarea handles all of them. This is not a compromise; it is a more honest representation of what these mechanics actually are.

4. State sync on every connection is the entire reconnection strategy
When a participant reconnects — WiFi hiccup, phone screen lock, accidental tab close — they receive current prices, round status, and leaderboard visibility immediately on connection. No reconnection-specific code path. The /display screen also recovers automatically if it is accidentally closed.

5. Send participants the URL 30 minutes before the event
A 150-person simultaneous login spike with credential confusion is the most common event-day crisis. Distributing the URL early and asking participants to verify their login beforehand converts this spike into a smooth 30-minute distribution. Credential issues surface before the event starts, not during it.

6. Optimistic UI makes trades feel instant
When a participant submits a trade, update the UI immediately before the server responds. Show the trade as pending. On server success, confirm. On failure, revert and show the inline error. On campus WiFi with 400-600ms latency to Render, without optimistic updates the interface feels broken. With them, it feels instant.

7. Admin sees a price preview before broadcasting
Before the admin clicks Broadcast on a news event, show the estimated new prices: "This will move VNTCH from 482 to approximately 554 (+15%)." This is two lines of JavaScript in the form handler. It prevents the most common admin error under event pressure: applying a shock in the wrong direction or to the wrong magnitude.

8. All prices in one object per tick
The price_update event carries { VNTCH: 482.5, VNDFN: 315.0, ... } for all tickers at once. Clients replace their entire price map each tick. This eliminates stale state bugs, makes portfolio computation trivial, and automatically includes any new IPO stock added mid-event without any changes to broadcast logic.

9. Leaderboard computed in memory, not on demand
The leaderboard runs alongside the price engine: recomputed every 30 seconds, stored in a server-side variable. When the admin clicks Reveal in front of 150 students, the response is instant. No database query at the high-attention moment.

10. One health check, one uptime monitor, one Telegram alert
GET /health returns 200. UptimeRobot monitors it every 5 minutes. If it goes down, you receive a Telegram notification within 5 minutes. Setup takes 10 minutes. This is the complete monitoring and alerting stack for a 4-hour event. You do not need Grafana. You need to know when the server is down.
